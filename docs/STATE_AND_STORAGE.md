# State and Storage

This document describes how different types of state are handled in the system and where each type of data should live.

Understanding where state belongs is critical for maintaining a predictable architecture.

The project separates state into several categories:

- UI state
- server state
- derived state
- URL state
- AI state

---

## UI State

UI state lives inside React Client Components.

Examples include:

- input values
- loading indicators
- error messages
- modal visibility
- temporary UI toggles

Example:

```text
const [loading, setLoading] = useState(false)
```

Characteristics of UI state:

- temporary
- local to a component
- reset on page refresh
- not persisted

UI state should never contain server data that must survive refresh.

---

## Server State

Server state is stored in the database and represents the **source of truth** for the system.

Examples include:

```text
posts
videos
AI metadata
moderation status
```

Server state is accessed through server services.

Typical flow:

```text
Server Component
→ server service
→ Prisma
→ database
```

Server state must never be duplicated in multiple places.

---

## Derived State

Derived state is computed from existing server data.

Examples:

```text
AI tags
AI confidence scores
moderation results
aggregated signals
```

Derived state may be:

- calculated during AI processing
- stored in the database for reuse
- recomputed when models improve

Derived state should not be recomputed unnecessarily.

If the computation is expensive, the result should be persisted.

---

## URL State

URL state is stored in query parameters.

Examples include:

```text
embed parameters
preview settings
user context
```

Example:

```text
/posts/5?userId=12
```

Properties of URL state:

- shareable
- bookmarkable
- stateless
- browser controlled

URL state should not contain sensitive information.

---

## AI State

AI state represents results generated by AI pipelines.

Two types of AI state exist.

### Quick AI State

Quick AI runs before publishing.

Characteristics:

- temporary
- not persisted
- used only to help the user preview results

Example signals:

```text
suggested hashtags
basic classification
anomaly hints
```

---

### Deep AI State

Deep AI runs after publishing and persists results to the database.

Examples of stored metadata:

```text
aiTags
aiConfidence
aiDescription
moderationStatus
moderationReason
```

Deep AI results enrich the post and may evolve as models improve.

---

## State Ownership Rule

Each type of state must have a clear owner.

```text
UI state → React client components
Server state → database
Derived state → AI pipeline or server services
URL state → browser URL
AI state → AI pipeline and persistence layer
```

Avoid duplicating the same data across multiple layers.

---

## Golden Rule

If the state must survive a page refresh → store it in the database.

If the state exists only to support UI interaction → keep it in React state.
